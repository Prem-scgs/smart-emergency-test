# Incident Status Tracking Design

Date: 2026-06-18
Status: Approved design, pending implementation plan

> Historical/spec-only note: this document preserves an earlier design. Endpoint
> examples such as `POST /api/incidents/:id/locations` should not be treated as
> current runtime API contract unless a later implementation explicitly adds
> them.

## Goal

Make Mobile incident history, location sharing, and case tracking use real backend data. Admin users update each case status, and Mobile receives the latest status and timeline in real time without replacing the existing incident-created alert flow.

## Scope

### Mobile

- Create an incident immediately when the user taps Call.
- Persist the selected contact phone as `dialed_phone` before opening the device dialer with `tel:`.
- Use a client-generated request ID and an idempotent create operation to prevent duplicate incidents.
- Keep the existing simulated call screen for the current demo flow. It may show the selected agency, dialed number, dialing state, and an end-call action.
- Do not treat simulated connection state or elapsed time as authoritative call data, and do not persist it as a real call result.
- Keep the call-result choices on the simulated call screen for the demo flow, but remove the reporter-phone form and validation.
- Submit the selected call result with `reporterPhone: null`, then open case tracking. Preserve and restore the active case if the device dialer backgrounds the web app.
- Show the latest status and complete status timeline.
- Keep incident history scoped to the existing reporter `sessionId`.
- Capture location when the incident is created.
- Allow the user to explicitly update their current location. Do not track location continuously in the background.

### Admin

- Use a three-column workspace: incident queue, map, and incident detail panel.
- Selecting a queue item focuses its map marker and loads the same incident into the detail panel.
- Selecting a marker selects the matching queue item.
- New incidents appear at the top with an unread indicator and alert sound, but do not replace the case currently being edited.
- Filter the queue by category, area, status, and search text.
- Reuse the incident detail panel from both Dashboard and Call Logs rather than creating a separate page for the MVP.

## Status Workflow

The ordered workflow is:

1. `reported` - แจ้งเหตุแล้ว
2. `acknowledged` - รับเรื่อง
3. `coordinating` - กำลังประสานงาน
4. `dispatched` - ส่งเจ้าหน้าที่
5. `on_scene` - ถึงที่เกิดเหตุ
6. `closed` - ปิดเหตุ

The backend supplies the timestamp for every transition.

### Role Rules

- `agency_admin` can view and update incidents in its own agency category only.
- `agency_admin` can move a case forward only.
- `super_admin` can view every incident and may move a case backward.
- Moving backward requires a reason.
- A resolution summary is optional when closing a case. If it is blank, the Admin UI must warn and require explicit confirmation before closing.
- Normal forward transitions do not require a note.
- Status history is append-only from application workflows.

## Data Model

### incidents changes

- Use `dialed_phone` as the immutable snapshot of the number selected when Call was tapped.
- Add `client_request_id` with a unique constraint for idempotent incident creation.
- Add `status_version` for optimistic concurrency control.
- Store the current workflow status in `incidents.status`.

### incident_status_history

Store one row per status transition:

- `id`
- `incident_id`
- `from_status`
- `to_status`
- `note`
- `changed_by_admin_id`
- `changed_by_role`
- `created_at` generated by the database/backend
- `status_version`

### incident_location_history

Store the initial location and every explicit Mobile update:

- `id`
- `incident_id`
- `latitude`
- `longitude`
- PostGIS point geometry
- `accuracy`
- `source` (`initial` or `user_update`)
- `created_at` generated by the database/backend

The latest accepted location also updates the incident location used by the Admin map.

## API Design

### Create incident and begin dialing

`POST /api/incidents`

The request includes `clientRequestId`, category, contact ID, `dialedPhone`, reporter session ID, and location. The backend creates the incident and initial history rows transactionally, then emits the existing `incident.created` SSE event.

Mobile sends the request with keepalive behavior and opens `tel:<dialedPhone>`. The stored value represents an attempted dial, not proof that the phone call connected.

The simulated call screen remains as a presentation layer for the current phase. It does not prove that the external phone call connected, and its timer or visual state must not update backend call facts.

### Read tracking data

`GET /api/incidents/:id/tracking`

Returns the incident, current status/version, status history, latest location, and location history. Access for Mobile requires the matching reporter session ID. Admin access uses the existing admin identity boundary, which must eventually be replaced by the team's JWT integration.

Mobile must call this endpoint when the tracking screen opens, when the user taps Refresh, and after reconnecting to realtime updates. Component-local snapshots may be used only as an immediate loading fallback and must be replaced by this authoritative response.

### Subscribe to one Mobile case

`GET /api/incidents/:id/events?sessionId=...`

The backend validates that the supplied reporter session owns the incident before opening the SSE stream. The stream emits only events for that incident, beginning with `incident.status_updated`; it must never expose the shared Admin event stream or another reporter's case.

When Mobile receives a status event, it refetches `GET /api/incidents/:id/tracking` rather than treating the event payload as the authoritative case state.

### Update status

`PATCH /api/incidents/:id/status`

Request:

- `fromStatus`
- `toStatus`
- `expectedVersion`
- optional `note`

The backend validates role scope and transition rules, then updates the incident and inserts status history in one transaction. A stale version returns `409 INCIDENT_STATUS_CONFLICT` with the current state so the Admin UI can reload.

### Update location

`POST /api/incidents/:id/locations`

Mobile sends the reporter session ID, coordinates, and accuracy. The backend verifies ownership, inserts location history, updates the incident location, resolves province/district through PostGIS, and emits a location update event.

## Realtime Events

Continue using SSE only.

- `incident.created` - Admin alert and queue insertion
- `incident.status_updated` - Admin synchronization and Mobile tracking update
- `incident.location_updated` - Admin map and Mobile location confirmation

Admin continues to use its role-scoped shared stream. Mobile uses one reporter-scoped stream for the case currently open on the tracking screen. Remove the duplicate active connection currently split between `use-sse` and the misleadingly named `use-websocket` implementation during implementation.

On reconnect, clients reload authoritative data through the API. SSE events are update signals, not the sole source of truth.

## Error Handling

- Duplicate `clientRequestId`: return the existing incident rather than creating another.
- GPS failure: prevent Call until a confirmed location is available, unless a separately approved fallback policy is introduced.
- Stale status version: return 409 and reload the current case without overwriting another Admin's update.
- Invalid role transition: return 403.
- Missing backward-transition reason: return 400 with a field-level error.
- SSE disconnect: retain the current screen, reconnect automatically, then refetch the selected case and queue.
- Invalid or mismatched Mobile reporter session: return 403 before opening the case stream.

## Testing

- Status transition matrix for both roles.
- Required-note rule for backward transitions, plus optional close-summary confirmation in the Admin UI.
- Concurrent Admin update conflict.
- Idempotent incident creation.
- Dialed phone snapshot persistence.
- Reporter session ownership for history, tracking, and location updates.
- Location history and PostGIS area resolution.
- SSE event visibility by role/category.
- Mobile case-stream ownership and cross-session rejection.
- Mobile opens, manually refreshes, and reconnects by refetching authoritative tracking data.
- Reconnect and refetch behavior.
- End-to-end flow: Mobile Call -> Admin queue -> Admin status update -> Mobile timeline.

## Explicitly Out of Scope

- Reading call connection state, duration, or call result from the device phone app.
- Persisting the simulated call screen's connection label or timer as real call data.
- Continuous background location tracking.
- WebSocket/WSS migration.
- A dedicated full-page incident route for the MVP.
- Implementing the team's final JWT provider. The feature must expose a clean authorization boundary for that integration.

## Approved Decisions

- Status timestamps use the current backend time.
- Only `super_admin` may move a status backward.
- Notes are optional for normal progress and case closure. Closing without a summary requires an explicit UI confirmation. Backward changes require a reason.
- Mobile opens tracking immediately after Call and can revisit cases through history.
- Location is captured initially and updated only when the user taps update.
- `agency_admin` updates its own category; `super_admin` updates all categories.
- The Admin incident queue is positioned to the left of the map.
- A new incident does not replace the case currently open in the detail panel.
- Concurrent updates use optimistic version checking rather than case locking.
- The simulated call screen remains temporarily with call-result options, without reporter-phone input, and routes to tracking after the result is submitted.
